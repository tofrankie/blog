## 前言

社区上用于管理 Node 版本的工具很多，比如：

- 非跨平台：[nvm](https://github.com/nvm-sh/nvm)、[n](https://github.com/tj/n)
- 跨平台：[nvs](https://github.com/jasongin/nvs)、[fnm](https://github.com/Schniz/fnm)、~~[volta](https://github.com/volta-cli/volta)~~（已停止维护）

此前用 nvm，现在切换到 fnm，原因是使用 nvm 时启动 Shell 太耗时。最直接的影响是那台 8GB 的 MacBook Pro 打开 VS Code 总会因为 Shell 解析太久导致 VS Code 终止解析，导致了某些扩展无法正常使用（[解决 Unable to resolve your shell environment in a reasonable time](https://github.com/tofrankie/blog/issues/47)）。另外，nvm 本身问题挺多的（[详见](https://github.com/nvm-sh/nvm#important-notes))。

## fnm 安装与使用

[fnm](https://github.com/Schniz/fnm) 基于 Rust 开发，跨平台。

> 🚀 Fast and simple Node.js version manager, built in Rust.

### 1. 安装 fnm

```bash
$ brew install fnm
```

### 2. 配置 Shell

```bash
$ fnm env --use-on-cd --shell zsh >> ~/.zshrc
```

也可执行 `fnm env --use-on-cd` 将输出内容手动添加至你的 Shell 配置文件里。

> 个人习惯使用 `fnm env --use-on-cd --shell zsh --corepack-enabled --resolve-engines`，一是安装 Node 时自动执行 `corepack enable`；二是自动检测 package.json 的 `engines.node` 选择/安装合适的 Node 版本。

### 3. 安装 Node

```bash
# LTS 版本
$ fnm install --lts

# 指定大版本的最新版本
$ fnm install 18

# 指定版本
$ fnm install 18.21.1
```

相反地，可通过 `fnm uninstall <version>` 或 `fnm uninstall <alias-name>` 来删除指定版本，后者会同时移除别名。

### 4. 切换 Node

```bash
# 使用系统版本
$ fnm use system

# 使用 fnm 所安装，且版本号为 18.21.1 的 Node 程序
$ fnm use 18.21.1

# 使用 fnm 所安装，且主版本号为 18 的最新版本的 Node 程序
$ fnm use 18
```
只要用 `fnm use <version>` 指定后，每次启动 Shell 将会默认使用对应的 Node 版本。

### 5. 设置别名

```bash
# 形式如：fnm alias <指定版本号> <别名>
$ fnm alias 18.21.1 v18

# 设置别名后，可以简化指令为：
$ fnm use v18
```

上述示例的别名意义不大，仅用于举例而已。直接使用 `fnm use 18` 就能切换至 `18.21.1` 版本。

比较有意义的是：

- `system`：以非 nvm 所安装的 Node，称为系统版本。比如通过官网或者 HomeBrew 安装等
- `default`：用于指定 `fnm` 所安装的 Node 默认版本

```bash
# 指定默认版本
$ fnm default 18.21.1

# 相当于
$ fnm alias 18.21.1 default
```

相反地，可通过 `fnm unalias <alias-name>` 来取消别名。

### 6. 项目中指定特定版本

在项目根目录下添加 `.node-version` 文件（兼容 `.nvmrc` 文件），并指定版本。比如：

```bash
$ echo '18' > .node-version
```

> 个人推荐用 `.nvmrc` 文件，因为 nvm 使用更广泛！

前提是配置 fnm 环境用 `fnm env --use-on-cd`，而不是 `fnm env`，后者没有添加 Hook，进入目录时不会是检查配置文件。对比输出就明白了。

在团队协作中，如果没有最低版本要求，个人推荐仅指定 major 版本，比如 `18` 而不是 `18.10`，这样团队成员即便是 minor 或 patch 版本不同都可以正常匹配，无需下载多个大版本相同的 Node.js。但如有版本要求，可以指定到更具体的 `18.10` 或 `18.10.1`，此时更合理的做法是在 `package.json` 中指定 `engines` 版本进行约束。

### 7. 卸载 fnm

若是通过 Homebrew 安装的 fnm，则：

```bash
$ brew uninstall fnm
```

接着，再移除 `~/.fnm` 目录。

```bash
$ rm -rf ~/.fnm
```

最后，移除 `bash` 或 `zsh` 的配置文件中与 `fnm` 相关的配置。比如：

```bash
export PATH="/Users/frankie/Library/Caches/fnm_multishells/49559_1670052262156/bin":$PATH
export FNM_VERSION_FILE_STRATEGY="local"
export FNM_DIR="/Users/frankie/Library/Application Support/fnm"
export FNM_NODE_DIST_MIRROR="https://nodejs.org/dist"
export FNM_MULTISHELL_PATH="/Users/frankie/Library/Caches/fnm_multishells/49559_1670052262156"
export FNM_ARCH="x64"
export FNM_LOGLEVEL="info"
autoload -U add-zsh-hook
_fnm_autoload_hook() {
  if [[ -f .node-version || -f .nvmrc ]]; then
    fnm use --silent-if-unchanged
  fi
}

add-zsh-hook chpwd _fnm_autoload_hook &&
  _fnm_autoload_hook

rehash
```

## nvm 与 fnm 原理

nvm 在每次启动 Shell 时，将对应 node 执行文件所在路径添加到 PATH 路径最前面。切换 Node 版本时，会更新此路径。

fnm 不太一样，它从固定路径（通过 fnm env 生成并写入到你的 Shell 配置文件里）去找 node 可执行文件。而这个路径是一个软链接，链接到 fnm 下载的对应的 Node 目录下。切换 Node 版本时，是更新这个固定路径的软链接指向。

对此，我觉得各有优缺点吧。

nvm 在不同 Shell 窗口 Node 版本不会相互影响，因为当前 Shell 的 PATH 是启动才注入的，不是持久化的，比如 A 项目的 Shell 切换 Node 版本，B 项目的 Shell 不会受影响。而 fnm 切换 Node 会影响所有已打开的 Shell 窗口。还有启动新 Shell 窗口时，fnm 不会切换回默认 Node 版本。

平时免不了要同时操作多个不同 Node 版本的项目，fnm 这种机制挺蛋疼的！

## nvm 启动慢的原因

nvm 不是传统的可执行文件，而是一组 Shell 函数，在 Shell 配置文件通过 `source nvm.sh` 加载它。

当 Shell 启动，它会：

1. 加载 `nvm.sh` 里所有函数定义
2. 执行 `nvm_process_paramters` 来处理自动版本切换
3. 根据配置自动使用或安装默认版本（内部使用了 `nvm use`），将对应版本的 node 路径放在 PATH 变量最前面，以确保 Shell 在搜索可执行文件时优先使用 nvm 的 node 二进制文件，而不是系统中其他位置的 node。

性能瓶颈出现在 `nvm use` 内部的 `nvm_ensure_version_installed` 过程（[详见](https://github.com/nvm-sh/nvm/blob/f695512c80308b3b98e6b3f4da8cf70ad472ccc8/nvm.sh#L4057-L4061)）。顾名思义，该操作涉及：

- 文件系统检查（查找已安装的版本）
- 可能的网络请求（如果需要查询远程版本）
- PATH、NVM_BIN、NVM_INC 等环境变量的修改

具体性能开销：

- 文件系统遍历：`nvm_version()` 需要扫描 `$NVM_DIR/versions/` 目录来查找匹配的版本
- 多次目录访问：函数需要检查多个位置（node、io.js、old 版本目录）
- 每次 Shell 启动都执行：这不是一次性操作，而是在每个新 Shell 会话启动时都要执行

使用 `source nvm.sh --no-use` 可以加快启动速度的原因是跳过了这个耗时的步骤。

## 移除 nvm

在移除之前，通过以下方式查看使用 `nvm` 所安装的全局包，然后切换到 `fnm` 安装一下（有需要的话）：

```bash
$ nvm use 16
Now using node v16.14.0 (npm v8.3.1)

$ npm list -g
/Users/frankie/.nvm/versions/node/v16.14.0/lib
├── corepack@0.10.0
├── npm@8.3.1
├── pnpm@7.5.0
├── simple-shell@
└── zx@7.0.7
```

移除 `nvm` 的安装目录，通常是 `~/.nvm`。执行以下命令即可：

```bash
$ rm -rf "$NVM_DIR"
```

移除 `bash` 或 `zsh` 的配置文件中与 `nvm` 相关的配置。比如：

```bash
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh" --no-use          # This loads nvm
[ -s "$NVM_DIR/bash_completion" ] && \. "$NVM_DIR/bash_completion" # This loads nvm bash_completion
```

其他系统可看 [Uninstalling / Removal](https://github.com/nvm-sh/nvm#uninstalling--removal)。

## 移除通过 pkg 安装的 Node.js

移除全局依赖以及 node、npm 等二进制软链接

```bash
$ sudo rm -rf /usr/local/lib/node_modules
$ sudo rm /usr/local/bin/node
$ rm /usr/local/bin/npm
$ rm /usr/local/bin/yarn
$ rm /usr/local/bin/pnpm
```

其他

```bash
$ sudo rm /usr/local/share/man/man1/node.1
```

与 Node.js 相关，但可能会影响其他形式安装的 Node.js 的 npm 所安装的依赖，移除请慎重！

```bash
$ sudo rm -rf ~/.npm
$ sudo rm -rf ~/.node-gyp
```
