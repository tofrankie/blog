## 背景

我现在使用 Codex 26.623.141536 版本。

项目使用的是 pnpm v10，但 Codex App（下称 Codex）内部却捆绑了 pnpm v11，这两个版本的依赖存储方式是不同的。因此如果执行 `pnpm add` 或 `pnpm install` 的时候需要重新生成依赖。这样会导致本地执行和 Codex 执行会有差异。

相关 Issue：[openai/codex#30440](https://github.com/openai/codex/issues/30440)

平时，我通常会使用 nvm 或 fnm 来管理 Node 版本，也会用 Homebrew 安装一个 Node 作为兜底，用于 VS Code Extension 等某些非交互式场景。总之，它们都配置在 `~/.zshrc` 里面，它只有交互式 Shell 才会加载，而 Codex 的 Agent 执行命令是在非交互式 Shell 进行的，所以无法拿到 nvm 或 fnm 这些命令。

Codex 的执行结果：

```bash
$ which node
/usr/local/opt/node@22/bin/node

$ which npm
/usr/local/opt/node@22/bin/npm

$ which yarn
/usr/local/opt/node@22/bin/yarn

$ which pnpm
/Users/frankie/.cache/codex-runtimes/codex-primary-runtime/dependencies/bin/pnpm
```

pnmp 使用了 Codex 内部的依赖，而不是我的全局 Node 22。

在 Codex 设置里有一个「Codex 依赖项」的选项（允许 Codex 安装并提供随附的 Node.js 和 Python 工具），尝试关掉也没有变化。

## 临时解决方案

在 `~/.zshenv` 里添加 Node 的路径 `export PATH="/usr/local/opt/node@22/bin:$PATH"`，但如果直接添加影响太大了，因为所有交互 Shell、非交互 Shell、脚本、git hook、其他程序等都会加载此配置。

所以添加一些限定范围：

```bash
# ~/.zshenv
if [[ ! -o interactive ]] && [[ "$TERM" = dumb ]] && [[ "$PATH" == *"/Users/frankie/.cache/codex-runtimes/codex-primary-runtime/dependencies/bin"* ]]; then
  export PATH="/usr/local/opt/node@22/bin:$PATH"
fi
```

当前 Node 22 用的是 pnpm v10。

**这种临时解决方案并不完美**。因为不同项目使用的 Node 版本或者 pnpm 版本要求不同，只是我目前的项目暂时未升级到 pnpm v11。

你也可以考虑把配置在 `~/.zshrc` 的 nvm 或 fnm 配置放到以上逻辑里，方便切换 Node 版本，但我是怕每次执行命令时因为 nvm 加载慢而影响整体响应速度，所以没加。
